# LiveKit 与实时语音配置

从这一版开始，TX-5DR 在 Docker、Linux 服务器版和桌面版上统一使用同一套 LiveKit 配置思路：

- 统一入口都是应用里的“系统设置 > 实时音频”
- 不再建议手工修改 `livekit.yaml`
- 系统会根据设置自动生成托管的 `livekit.resolved.yaml`
- 如果你的部署方式本质上只能暴露网页入口而不能打通媒体端口，应直接使用 `ws-compat`

如果你只记住一句话：

> 能稳定暴露 LiveKit 媒体端口，就用 LiveKit；只能暴露网页或 `/livekit` 路径，就用 `ws-compat`。

## 先分清两个地址

LiveKit 经常让人困惑，是因为它至少涉及两类“地址”，而且它们不是一回事。

### 1. 浏览器入口 / signaling 地址

这是浏览器最先连接的地址，通常表现为当前站点下的同源 `/livekit` 路径，或者你在系统设置里填写的“自定义实时语音入口”。

它解决的是：浏览器先去哪里拿房间和信令。

### 2. 媒体地址 / 媒体端口

这是 WebRTC 真正传音频时要连到的地址，通常依赖：

- `7881/tcp`
- `50000-50100/udp`

它解决的是：音频包最终应该发到哪台机器、哪个端口。

::: warning 只反代 `/livekit` 不等于 LiveKit 一定可用
如果你只是把网页入口、域名，或者 `/livekit` 这条 signaling 路径转发了出去，但没有同时打通 LiveKit 的 TCP/UDP 媒体端口，浏览器往往会卡在连接中，或者直接 ICE 失败。
:::

这类场景下，请优先改用 `ws-compat`。它只依赖站点本身的单一入口，更适合：

- FRP 只转网页
- 只有一个可暴露端口
- 只做 HTTPS 反代，不方便额外开 UDP
- 完全无法确定真正公网出口 IP

## 统一配置入口

无论你使用哪种分发形态，推荐都按下面顺序配置：

1. 打开“系统设置 > 实时音频”
2. 先决定传输策略：`Auto`、`LiveKit` 或 `ws-compat`
3. 如果使用 LiveKit，再选择“LiveKit 媒体网络模式”
4. 仅在“公网手动指定”时填写公网 IPv4
5. 保存后按对应部署方式重启 LiveKit 或应用

其中最常用的两组设置是：

| 你的实际场景 | 推荐设置 |
| --- | --- |
| 本机测试、桌面本机、同一局域网 | `Auto` 或 `LiveKit` + `局域网 / 本机` |
| 云主机、真实公网 IP、自动探测可靠 | `Auto` 或 `LiveKit` + `公网自动发现` |
| 家宽 NAT、多网卡、自动探测经常选错地址 | `Auto` 或 `LiveKit` + `公网手动指定` |
| FRP 只转网页、只反代 `/livekit`、只开单端口 | 直接用 `ws-compat` |

## 三种 LiveKit 媒体网络模式怎么选

### 局域网 / 本机

适合：

- Electron 桌面本机使用
- Docker 或 Linux 服务器只在同一局域网访问
- 开发调试、本地联调

该模式不会向外宣告公网媒体地址。对于本机或 LAN 场景，它通常是最直接、最稳定的选择。

### 公网自动发现

适合：

- 云服务器
- 有真实公网 IP 的主机
- STUN 自动探测结果稳定可靠的环境

该模式会让 LiveKit 自动推断应向浏览器宣告哪个公网地址。对于标准云主机，这通常是最省心的选择。

### 公网手动指定

适合：

- 家宽 NAT
- 有多个网卡或多个出口
- 自动探测经常选错地址
- 你已经明确知道浏览器真正应该访问哪个公网 IPv4

该模式要求你在系统设置里填写“手动媒体公网 IPv4”。

::: danger 这里填的不是 `/livekit` 的网址
这里填写的是浏览器建立 WebRTC 媒体连接时要使用的公网 IPv4，而不是网页访问地址，也不是域名路径本身。
:::

## Docker、Linux、桌面版如何统一理解

三种部署方式虽然打包方式不同，但配置思路完全一样：

- 浏览器入口从系统设置里配
- LiveKit 媒体模式从系统设置里选
- 运行文件由系统自动生成
- 保存后重启对应运行单元

### Docker

- Compose 现在默认把整个 `./data` 根目录持久化到容器内的 `/app/data`
- LiveKit 运行文件由系统生成到 `./data/realtime/livekit.resolved.yaml`
- 不要手工修改这个生成文件；镜像更新、初始化脚本或设置保存后都可能重新生成
- 改完设置后，至少重启 LiveKit 容器一次

常见操作：

```bash
docker compose -f docker-compose.livekit.yml restart livekit
```

如果你刚改完整套部署、环境变量或反向代理，也可以直接重启整个栈：

```bash
docker compose -f docker-compose.livekit.yml up -d
```

### Linux 服务器版

- 安装包会生成并维护 `/var/lib/tx5dr/realtime/livekit.resolved.yaml`
- 这同样是托管运行文件，不建议手工编辑
- 改完“系统设置 > 实时音频”后，执行一次服务重启即可

推荐命令：

```bash
sudo tx5dr restart
```

如果你只想单独处理 LiveKit，也可以使用：

```bash
sudo systemctl restart tx5dr-livekit
sudo systemctl restart tx5dr
```

### 桌面版（Electron）

- 桌面版使用相同的设置页和相同的三种网络模式
- 运行时同样会生成托管的 `livekit.resolved.yaml`
- 更改后通常重启整个应用最省心

桌面版最常见的选择通常是：

- 只在本机窗口使用：`局域网 / 本机`
- 让同一局域网里的浏览器访问桌面机：通常仍然是 `局域网 / 本机`
- 通过公网或复杂代理让外部浏览器访问桌面机：优先评估是否应该改用 Linux 服务器版；如果只是网页反代或隧道转发，建议直接用 `ws-compat`

## FRP、反向代理、家庭宽带的推荐策略

这几类场景最容易踩坑，可以直接按下面判断：

### FRP 只转网页 / HTTPS

推荐：`ws-compat`

原因：

- 浏览器入口能通，不代表 WebRTC 媒体能通
- FRP 常见配置只转单个 TCP 端口，不会自动处理 LiveKit 需要的 UDP 范围
- 真实公网出口 IP 往往不是系统在本机内能准确知道的那个值

### Nginx / Caddy 只反代网站和 `/livekit`

推荐：

- 如果也能正确暴露 `7881/tcp` 和 `50000-50100/udp`，可以继续用 LiveKit
- 如果不能，就改用 `ws-compat`

### 家宽 NAT，但你知道公网 IPv4，且能做端口映射

推荐：`公网手动指定`

前提是你已经：

- 在路由器上映射了 `7881/tcp`
- 放通了 `50000-50100/udp`
- 明确知道浏览器应访问的公网 IPv4

### 云服务器直出公网

推荐：`公网自动发现`

如果自动结果不对，再切换到 `公网手动指定`。

## 可靠运行检查清单

部署完成后，按这个顺序检查最稳妥：

1. 浏览器是否通过 HTTPS 或 `localhost` 访问
2. 如果用 LiveKit，媒体端口是否真的放通
3. 如果用反向代理，`/livekit` 是否只是 signaling，而不是全部媒体通道
4. 如果你无法稳定判断公网出口 IP，是否应该改用 `ws-compat`
5. 保存设置后，是否已经重启 LiveKit 或应用

## 不建议再做的事

以下做法现在都不推荐：

- 直接手改 `livekit.yaml`
- 把 `/livekit` 地址误当成媒体公网 IP
- 只看网页能打开，就认为 LiveKit 一定能传音频
- 在 FRP 只转网页的前提下坚持使用 LiveKit
